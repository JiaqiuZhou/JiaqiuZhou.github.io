主要写一些从课堂上获取的关于软件架构的一些知识

1. 软件架构的概念

1.1 什么叫做软件？

  • 首先是计算机技术，对于计算机的技术，每年都在日益更新，但是所有基本原理都会是一样的，因为计算机硬件架构的局限性，使其变化也相应的被限制
  • 然后是工程管理，涉及到实际工程部署，架构设计，整个软件开发流程
  • 最后是领域知识，包括数学知识，比如说在一个特定的银行进行app开发，懂金融的人会更加友好

1.2 什么是软件架构？

  • 从软件发展的历史来看，从高级语言到面向对象(可执行程序复用), 到构件(DLL系统复用), 到服务(企业网络复用), 再到语义服务(不同企业之间动态可配置)

  • 架构是组件与组件之间的联系，组件与环境之间的交互的设计准则，其应用场景可以如下所示：

    从应用环境—>根据环境设计多种架构(如分布式架构等)—>应用组件—>组件优化

1.3 新的软件架构描述

  • 模型驱动架构: 为了更加友好的在更换需求的时候不用再从底层开始修改代码,应该将架构设计为模型+引擎的方式, 如HTML与浏览器,即修改需求的时候只需修改模型,而无需修改程序
  • 举例来说的话, 12306购票的时候,如果每个人都去访问数据库,那么服务器一下子就炸掉了. 这里就需要通过软件架构对其进行设计,通过分布式的服务器或者说一个总线方式连接,这样就会满足更多的需求

2. 架构的演化过程

  • 单体: Database Server (1)—> Application Server(2) —> Browser PC Client(3)
  • 纵向分解: 多个单体并排
  • 分布式计算: 即单体的三个组成部分,在单体并排的时候,其两个两个之间可以互相调用自己的组成部分,如第二个单体的2部分可以调用第一个单体的1部分
  • SOA架构(ESB): 通过将组成部分都放到总线上的这样一种方式,来进行通讯交流,但是其可靠性并不能得到保证
  • 纵向化—>横向化:架构功能分解\数据分解\复制载荷均衡\平台相关模型\平台无关模型

3. 软件工程

3.1 具备能力

  • 解析能力:问题剖析能力
  • 系统性:即全局观察能力
  • 设计能力, 设计方式:数学分析,模型建立能力(数学/物理/经济),领域知识(业务模型)
  • 编程能力

3.2 程序员—>架构师

  • 能不能在设计阶段就判断出设计性能:比如网络系统,设计完就能判断多少人进去会瘫痪

  • 面向对象方法学:系统开放设计方法,真正做到代码与文档同步

    3.3 时代方向

  • 当今情况下,操作系统和中间键的团队少之又少,都是基于上层在做,导致在于其他国家进行竞争的时候根本就没有软件的东西可以拿出来干.处于非常被动的状态

4. 其他收获

  • java的开发并不是为了一种编程语言的诞生而诞生,而是美国军方当时为了设计一种能够部署在小型硬件上面给月球车等月球这样一种方便沟通的语言才诞生的,然后后来人们发现用在浏览器上很方便,所以才成为了编程语言

  • 所以说很多事情的演化过程可能就是自然而然的,做好每一步的努力,脚踏实地即可.

  • 从另一个角度看阿里双十一，不仅是一个狂欢盛宴，更是一个大型的测试调试现场
